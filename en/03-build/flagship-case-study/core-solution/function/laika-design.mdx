  **Purpose:** This document defines the **internal** operational logic of the Design Management `üìã Function`. It manages the entire creative process, from initial concept development to final client approval and the handover of technical drawings to the project management team.
</Info>

## Why This Matters

This `üìã Function` provides a structured and collaborative platform for the entire design lifecycle, ensuring that client feedback is systematically captured and design versions are meticulously tracked. It centralizes all design files, feedback, and approvals, preventing miscommunication and the use of outdated files. Strategically, this accelerates the approval process, creates a clear audit trail of design decisions, and ensures the final, approved designs are seamlessly integrated into the project execution plan. 

## The Big Picture

Architecturally, this `üìã Function` operates in parallel with the early stages of the Project Management `üìã Function`. It is built upon an extensive hierarchical data model, starting with the `üßä Design Drawing` as the main container, which links to a parent `üßä Project`. This container is broken down through four levels of sub-objects: `üßäÔ∏é Design Category`, `üßäÔ∏é Design Space`, `üßäÔ∏é Item in Space`, and finally, `üßäÔ∏é File Version`. This deep structure allows for granular control and tracking of every single design element, from a high-level 3D render down to a specific version of a cabinet drawing.

## OBJECT TYPES & WORKFLOWS

### 1. üßä Design Drawing (Standard Object)

- **Process Description**: The top-level container for a complete design package for a client project. It groups all related design categories, spaces, and file versions.
- **Example**: "DD-101: Design Package for Hoai An's Apartment".

<Tabs>
  <Tab title="Workflow">
    **Design Drawing Workflow**

    `INITIATED` ‚Üí `DESIGNING` ‚Üí `AWAITING FEEDBACK` ‚Üí `FINALIZED` ‚Üí `HANDED OVER`
  </Tab>
  <Tab title="Status Details">
    | Status               | Business Logic                                                                                           | Advanced Settings                                                                                    |
    | :------------------- | :------------------------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------- |
    | 1. INITIATED         | The design brief has been received and the design package is being set up.                               | **Automation**: Creates the default framework of `Design Category` and `Design Space` sub-objects.   |
    | 2. DESIGNING         | The design team is actively working on creating the concepts and drawings.                               | **Input Screen**: Designers upload `File Version` sub-objects.                                       |
    | 3. AWAITING FEEDBACK | A version has been sent to the client and the team is waiting for review and comments.                   | **Automation**: Sends a notification to the Client that a new version is available for review.       |
    | 4. FINALIZED         | The client has approved the overall design plan. No more creative changes are expected.                  | **Condition**: All child `Design Category` objects must be in a "Completed" status.                  |
    | 5. HANDED OVER       | The finalized set of technical drawings has been delivered to the Project Management team for execution. | **Automation**: Copies the approved files to the related `Project`'s "Construction Drawings" folder. |
  </Tab>
  <Tab title="Data Fields">
    | Group                 | Field Name (Type)                                                                                     |
    | :-------------------- | :---------------------------------------------------------------------------------------------------- |
    | **Basic Information** | Design Code (Text), Related Project (Object Reference), Design Style (Dropdown), Lead Designer (User) |
    | **Version Tracking**  | Current Version (Number), Last Feedback Date (Date), Approval Status (Text)                           |
  </Tab>
</Tabs>

### 2. üßäÔ∏é Design Category (Sub-Object)

- **Process Description**: A high-level grouping of design deliverables, such as 3D visualizations or 2D technical drawings.
- **Parent Object**: `üßä Design Drawing`

<Tabs>
  <Tab title="Workflow">
    `NOT STARTED` ‚Üí `IN PROGRESS` ‚Üí `COMPLETED`
  </Tab>
  <Tab title="Status Details">
    | Status         | Business Logic                                                           |
    | :------------- | :----------------------------------------------------------------------- |
    | 1. NOT STARTED | Planning for this category is complete, but work has not begun.          |
    | 2. IN PROGRESS | The team is actively creating the deliverables for this category.        |
    | 3. COMPLETED   | All work for this category is finished and has been internally reviewed. |
  </Tab>
  <Tab title="Data Fields">
    | Group                | Field Name (Type)                                                  |
    | :------------------- | :----------------------------------------------------------------- |
    | **Category Details** | Category Type (Dropdown), Person in Charge (User), Due Date (Date) |
  </Tab>
</Tabs>

### 5. üßäÔ∏é File Version (Sub-Object)

- **Process Description**: Manages an individual file submission for a specific design item, tracking its review and approval workflow.
- **Parent Object**: `üßäÔ∏é Item in Space`
- **Example**: "TV Cabinet - v1_draft.png", "TV Cabinet - v2_final.pdf".

<Tabs>
  <Tab title="Workflow">
    `SKETCH` ‚Üí `PENDING REVIEW` ‚Üí `APPROVED` / `REJECTED`
  </Tab>
  <Tab title="Status Details">
    | Status            | Business Logic                                                       | Advanced Settings                                                                        |
    | :---------------- | :------------------------------------------------------------------- | :--------------------------------------------------------------------------------------- |
    | 1. SKETCH         | A new design file or revision is being prepared for submission.      | **Input Screen**: Upload file, add notes.                                                |
    | 2. PENDING REVIEW | The file has been submitted to the client or a manager for feedback. | **Permissions**: Client & Manager. **Automation**: Sends a notification to the reviewer. |
    | 3. APPROVED       | The file has been officially approved.                               | **Automation**: Updates the status of the parent `Item in Space`.                        |
    | 4. REJECTED       | The file has been rejected and requires revision.                    | **Input Screen**: Reason for rejection.                                                  |
  </Tab>
  <Tab title="Data Fields">
    | Group            | Field Name (Type)                                                                              |
    | :--------------- | :--------------------------------------------------------------------------------------------- |
    | **File Details** | File (Attachment), Version Number (Number), Submission Date (Date), Feedback Notes (Paragraph) |
  </Tab>
</Tabs>

## OBJECT RELATIONSHIPS

| Relationship                                | Type         | Description                                                     |
| :------------------------------------------ | :----------- | :-------------------------------------------------------------- |
| `üßä Design Drawing` ‚Üí `üßäÔ∏é Design Category` | Parent-Child | A design package is split into major categories (e.g., 3D, 2D). |

| `üßäÔ∏é Item in Space` ‚Üí `üßäÔ∏é File Version`    | Parent-Child | Each item has one or more file versions tracking its progress.  |
| `üßä Design Drawing` ‚Üí `üßä Project`          | Reference    | Each design package is linked to one master project.            |

## AUTOMATIONS

| Rule                                | Trigger                                                    | Action                                                                                                                                   |
| :---------------------------------- | :--------------------------------------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------- |
| 1. **Create Default Structure**     | A `üßä Design Drawing` is created.                          | Automatically creates a standard set of sub-objects: `üßäÔ∏é Design Category` (3D, 2D) and `üßäÔ∏é Design Space` (Living Room, Bedroom, etc.). |
| 2. **Notify Client of New Version** | A `üßäÔ∏é File Version` is moved to `PENDING REVIEW`.         | Sends an automated email or portal notification to the client with a link to view the file and provide feedback.                         |
| 3. **Update Parent Status**         | All `üßäÔ∏é File Version` objects for an item are `APPROVED`. | Automatically changes the status of the parent `üßäÔ∏é Item in Space` to `COMPLETED`. This rolls up the hierarchy.                          |

## ROLES, PERMISSIONS & NOTIFICATIONS

<Tabs>
  <Tab title="Roles">
    | Role               | Description                                                                                                   |
    | :----------------- | :------------------------------------------------------------------------------------------------------------ |
    | **Design Manager** | Responsible for approving final designs, managing the design team's workload, and ensuring quality standards. |
    | **Designer**       | Responsible for creating, revising, and submitting all design files and versions.                             |
    | **Client**         | Responsible for providing timely feedback on submitted designs and giving final approval on `File Versions`.  |
  </Tab>
  <Tab title="Permissions">
    This `üìã Function` uses **Scheme 4 - Extended Permissions**.

    - **`View ‚èπÔ∏è Space`**: All users with a role in the Space.
    - **`View & Update üßä Object`**: All users who can view the Space.
    - **`Create üßä Object`**: `Design Manager` and `Designer` roles only.
    - **`Delete üßä Object`**: `Design Manager` role only.
  </Tab>
  <Tab title="Notifications">
    This `üìã Function` uses **Scheme 3 - Balanced Notifications**.

    - **Object Updates**: Notifications for creating or updating an Object are sent only to the assignee and the person who created the Object.
    - **Messages**: Chat messages sent within an Object will notify all users who are following that Object.
  </Tab>
</Tabs>
